## 2. Backend Code Patterns

### 2.1 Command Parsing — Timezone Handling (`packages/backend/src/agent/parse-command.ts`)

The code defines a **hardcoded timezone constant**:

```typescript
// Line 74
const TIMEZONE = config.timezone ?? "Asia/Ho_Chi_Minh";
```

This is used to:
- Parse user input times (e.g., "gửi lúc 8h sáng") as Asia/Ho_Chi_Minh local time
- Convert parsed local times to UTC via `toUtc()` (line 76-84)

The `toUtc()` function parses a local ISO string and returns `.toISOString()` (UTC):

```typescript
function toUtc(localISO: string): string {
  try {
    const d = new Date(localISO);   // Parses as local ISO → UTC
    return d.toISOString();
  } catch {
    return localISO;
  }
}
```

**⚠️ ISSUE:** `new Date(localISO)` in Node.js interprets ISO strings as UTC. But the `localISO` values being fed to this function are generated by `parseTime()` and `partOfDay()`, which use `nowInTimezone()` (UTC+7 via offset) to construct local dates, then call `.toISOString()` on them — which outputs UTC. So `toUtc()` receives an **already-UTC** string and re-parses it, effectively being a no-op. **This works correctly by accident** but the logic is confusing.

The `nowInTimezone()` function (line 86-90) uses a **manual offset approach**:

```typescript
function nowInTimezone(): Date {
  const utc = new Date();
  return new Date(utc.getTime() + 7 * 60 * 60 * 1000);
}
```

**⚠️ ISSUE:** This creates a Date object that is 7 hours ahead of UTC, but Node.js always displays Dates in UTC via `.toISOString()`. The returned Date object represents a UTC timestamp that's +7h offset. When `.toISOString()` is called on it, it returns a UTC time that is 7 hours in the future. This is used in `parseTime()` (line 200-208) to check "is this time in the past?" — which works fine since both `now` and `target` are offset by the same amount.

**Overall assessment:** The scheduling timezone logic functions correctly but is **fragile** — it relies on a manual offset rather than proper timezone-aware date handling. The `TIMEZONE` constant is effectively unused (it's set but intellisense/Intl.DateTimeFormat is only used in the frontend).

## 4. LiveTest `expiresAt` — Potential Misleading Display

### Backend (`live-test.service.ts`)

```typescript
// Session creation (line 139-140)
const now = new Date();  // UTC on this server
const expiresAt = new Date(now.getTime() + ttlSeconds * 1000);

// Status response (line 244)
expiresAt: session.expiresAt.toISOString(),  // UTC ISO string
```

### Frontend (`production-readiness/page.tsx`)

```typescript
// Line 201 — TTL is shown as seconds remaining using remainingMs
<span>TTL: {Math.round(ltStatus.session.remainingMs / 1000)}s còn lại</span>
```

```typescript
// Line 202 — createdAt shown with formatVnTime
<span>Tạo lúc: {formatVnTime(ltStatus.session.createdAt)}</span>
```

**Analysis:** The `remainingMs` is calculated server-side (line 235):
```typescript
const remainingMs = Math.max(0, session.expiresAt.getTime() - Date.now());
```

Both `session.expiresAt.getTime()` and `Date.now()` use the same UTC clock, so the calculation is correct. ✅

**However**, `expiresAt` itself is NOT displayed on the frontend. Only:
- `remainingMs` (as seconds)
- `createdAt` (as Vietnam time)

**Result:** `expiresAt` is NOT shown to users. If it were displayed, it would appear via `formatVnTime()` as UTC+7 — which would be correct. **No misleading behavior found.** ✅

## 6. Recommendations

1. **Set server timezone to UTC+7** (low priority): Set `TZ=Asia/Ho_Chi_Minh` in the server environment or PM2 ecosystem config. This would make `date` output and cron jobs align with Vietnam time. Not critical since all logic uses UTC internally.

2. **Refactor `nowInTimezone()` to use proper timezone API** (medium priority): Replace the manual +7h offset with `Intl.DateTimeFormat` or a library like `luxon` for clarity and future-proofing. Current code works only because Vietnam has no DST.

3. **Add `expiresAt` display to LiveTest UI** (low priority): Showing the actual expiry time (in VN time) next to the countdown would improve UX. Currently only `createdAt` and `remainingMs` are shown.

4. **Document the timezone contract**: Add a note in `AGENTS.md` or architecture docs stating: *"All backend timestamps are UTC ISO 8601. Frontend converts to Asia/Ho_Chi_Minh for display and labels as UTC+7."*
